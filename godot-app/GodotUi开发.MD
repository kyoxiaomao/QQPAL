## 一、文档目标

这份文档用于定义 QQPAL 简单版桌宠前台在 Godot 中的开发范围、UI 结构、交互方式和状态机边界，目标不是一次做完整产品，而是先做出一个可以演示、可以继续扩展的最小版本。

这一版重点解决四件事：

- 先做 **1 只桌宠**
- 先做 **1 套桌宠状态机**
- 先做 **左键简洁对话框 + 右键菜单**
- 先做 **应用主界面入口**

---

## 二、本期范围

### 1. 要做的内容

- 一只常驻桌面的 Godot 桌宠
- 桌宠基础状态切换与表现映射
- 左键点击桌宠后弹出简洁对话框
- 右键点击桌宠后弹出操作菜单
- 菜单中包含“打开应用”核心入口
- 打开应用后进入完整主界面
- 主界面中预留对话、模型配置、技能配置、系统设置等页面

### 2. 暂不做的内容

- 多桌宠
- 复杂战斗式或游戏式交互
- 完整商城、登录、商业化后台
- 复杂动画状态混合树
- 复杂语音系统
- 正式生产环境安装器和自动更新机制

### 3. 本期目标

本期只要求做到：

> 用户能在桌面看到桌宠，左键能快速聊天，右键能打开菜单，菜单能进入主应用，主应用里能管理系统相关配置，并且桌宠状态能随系统任务状态变化。

---

## 三、版本约束

### 1. 引擎版本

- 使用 **Godot 4.6 稳定版**

### 2. 开发原则

- 先做最小闭环，再做美术升级
- 先保证结构清晰，再补细节特效
- 桌宠表现层与业务状态层分离
- UI 页面可先用占位内容，不追求首版视觉完成度

### 3. 本期推荐技术方向

- Godot 负责前台表现、交互和状态展示
- Runtime 负责本地状态、任务进度和配置落地
- 云主 Agent 负责任务理解与编排
- Godot 先通过统一状态字段消费外部结果，不直接承担复杂业务逻辑

## 四、交互定义

这一版桌宠只定义三类核心交互：悬停、左键、右键。

### 1. 悬停

- 桌宠高亮或播放轻微待机反馈
- 可显示一句简短提示
- 不弹出复杂面板

### 2. 左键点击桌宠

左键行为定义为：

> 打开简洁版对话框。

这个对话框只承担轻量交互，不承担完整设置管理。

简洁版对话框建议包含：

- 输入框
- 发送按钮
- 最近 3 到 6 条对话展示
- 当前状态提示
- 一个“展开应用”按钮

适合承载的内容：

- 快速问一句话
- 查看当前是否空闲
- 查看最近一次任务摘要
- 快速跳转到完整应用

不建议放入的内容：

- 大量配置表单
- 深层模型参数编辑
- 大量历史记录
- 复杂多标签页

### 3. 右键点击桌宠

右键行为定义为：

> 打开桌宠快捷菜单。

菜单建议首版包含以下项目：

- 打开应用
- 快速对话
- 查看状态
- 暂停/恢复桌宠
- 退出

其中最核心的一项是：

- **打开应用**

“打开应用”用于进入完整主界面，是所有复杂设置和功能的主入口。

## 五、界面结构

这一版建议将 Godot 前台拆成两层界面：

- **桌宠浮层**
- **主应用层**

### 1. 桌宠浮层

桌宠浮层负责轻量交互，特点是轻、快、始终可达。

包含模块：

- 桌宠实体
- 右键快捷菜单
- 左键简洁对话框
- 状态气泡
- 少量提示反馈

### 2. 主应用层

主应用层负责完整管理能力，特点是信息更全、操作更完整。

包含模块：

- 首页 / 总览页
- 对话界面
- 模型配置
- 技能配置
- 系统设置
- 状态与日志页

### 3. 两层关系

- 桌宠浮层用于高频入口
- 主应用层用于完整操作
- 左键优先进入轻交互
- 右键优先进入快捷操作
- 所有复杂设置最终都收敛到主应用层

## 六、桌宠状态机

### 1. 状态机目标

桌宠状态机用于把系统内部状态映射为用户可感知的桌宠表现。

这版先不要做复杂状态树，只做一套清晰的有限状态机。

### 2. 首版状态定义

建议桌宠首版至少包含以下状态：

| 状态名 | 含义 | 表现建议 |
| --- | --- | --- |
| idle | 空闲待机 | 普通待机动作、轻微呼吸感 |
| hover | 鼠标悬停 | 抬头、提示气泡、高亮 |
| talking | 左键对话中 | 面向对话框、轻聊天反馈 |
| planning | 正在思考任务 | 思考动作、转圈、提示“思考中” |
| running | 正在执行任务 | 忙碌动作、状态条或提示 |
| waiting_user | 等待用户确认 | 提醒动作、问号、提示气泡 |
| success | 任务成功 | 开心动作、短暂庆祝反馈 |
| failed | 任务失败 | 失落动作、错误提示 |
| paused | 桌宠暂停 | 灰化、低活跃表现 |

### 3. 状态来源

状态来源分两类：

- **本地交互状态**：hover、talking、paused
- **业务映射状态**：planning、running、waiting_user、success、failed

### 4. 状态优先级

为了避免状态冲突，建议首版按以下优先级覆盖：

`failed > waiting_user > running > planning > talking > hover > paused > idle`

如果后续需要，也可以把 `paused` 提到更高优先级，但首版建议先按交互友好度处理。

### 5. 状态流转示例

#### 示例 A：普通交互

`idle -> hover -> talking -> idle`

#### 示例 B：任务执行

`idle -> talking -> planning -> running -> success -> idle`

#### 示例 C：任务等待补充

`idle -> talking -> planning -> waiting_user -> talking -> running -> success`

#### 示例 D：任务失败

`idle -> talking -> planning -> running -> failed -> idle`

### 6. 表现层建议

首版不要求复杂骨骼动画，可以先按以下顺序实现：

- 第一阶段：不同状态切换不同贴图或简单动画
- 第二阶段：补帧动画或 SpriteFrames
- 第三阶段：再升级成更复杂的骨骼或 2D 动画方案

## 七、场景、节点与资源目录规划

这一版建议先按可维护性拆成几个独立场景，而不是把所有 UI 堆在一个场景里。

### 1. 推荐场景结构

```text
godot-app/
├─ project.godot
├─ scenes/
│  ├─ app/
│  │  ├─ MainApp.tscn
│  │  ├─ pages/
│  │  │  ├─ HomePage.tscn
│  │  │  ├─ ChatPage.tscn
│  │  │  ├─ ModelConfigPage.tscn
│  │  │  ├─ SkillConfigPage.tscn
│  │  │  └─ SystemSettingsPage.tscn
│  ├─ pet/
│  │  ├─ DesktopPet.tscn
│  │  ├─ PetBubble.tscn
│  │  ├─ PetContextMenu.tscn
│  │  └─ QuickChatPanel.tscn
│  └─ common/
│     ├─ ConfirmDialog.tscn
│     └─ StatusBadge.tscn
├─ scripts/
│  ├─ pet/
│  ├─ app/
│  ├─ state/
│  └─ services/
└─ assets/
   └─ pets/
      ├─ textures/
      ├─ portraits/
      ├─ effects/
      └─ animations/
         ├─ idle/
         ├─ hover/
         ├─ talking/
         ├─ planning/
         ├─ running/
         ├─ waiting_user/
         ├─ success/
         ├─ failed/
         └─ paused/
```

### 2. 资源目录规范

为了避免后续动画资源越放越乱，桌宠相关资源建议从第一版开始就按“角色 / 类型 / 状态”分层。

推荐目录如下：

```text
godot-app/
└─ assets/
   └─ pets/
      ├─ textures/
      ├─ portraits/
      ├─ effects/
      └─ animations/
         ├─ idle/
         ├─ hover/
         ├─ talking/
         ├─ planning/
         ├─ running/
         ├─ waiting_user/
         ├─ success/
         ├─ failed/
         └─ paused/
```

这一版你现在最先会用到的帧动画目录建议固定为：

- `idle`：`godot-app/assets/pets/animations/idle/`
- `hover`：`godot-app/assets/pets/animations/hover/`
- `talking`：`godot-app/assets/pets/animations/talking/`
- `running`：`godot-app/assets/pets/animations/running/`

文件命名建议统一为：

- `idle_01.png`
- `idle_02.png`
- `hover_01.png`
- `talking_01.png`
- `running_01.png`

当前这轮实际开发已存在的 idle 资源命名为：

- `frame_000001.png`
- `frame_000002.png`
- `frame_000003.png`
- `...`

因此首版工程实现先以现有真实资源命名为准，不强制重命名；桌宠场景运行时直接从 `assets/pets/animations/idle/` 目录读取序列帧。

这样做的好处是：

- 后续状态扩展时不需要重整资源目录
- `SpriteFrames` 导入时更容易按状态批量处理
- 程序侧读取路径时更直观

### 3. DesktopPet 场景建议

`DesktopPet.tscn` 建议承担：

- 桌宠显示
- 鼠标点击响应
- 桌宠状态切换
- 呼出简洁对话框
- 呼出右键菜单

建议节点关系如下：

```text
DesktopPet
├─ PetVisual
├─ ClickArea
├─ AnimationPlayer
├─ StateController
├─ PetBubble
├─ QuickChatPanel
└─ PetContextMenu
```

### 4. MainApp 场景建议

`MainApp.tscn` 建议承担完整应用窗口。

建议节点关系如下：

```text
MainApp
├─ TopBar
├─ SideNav
├─ PageContainer
│  ├─ HomePage
│  ├─ ChatPage
│  ├─ ModelConfigPage
│  ├─ SkillConfigPage
│  └─ SystemSettingsPage
└─ GlobalDialogLayer
```

### 5. 节点职责建议

- `DesktopPet`：桌宠主控制器
- `PetVisual`：角色视觉表现
- `ClickArea`：左键、右键、悬停判定
- `StateController`：状态流转管理
- `QuickChatPanel`：简洁版对话框
- `PetContextMenu`：桌宠右键菜单
- `MainApp`：完整主应用窗口

## 八、UI 页面规划

### 1. 简洁版对话框

这是左键点击桌宠后的轻量面板。

建议包含：

- 顶部状态标题
- 对话消息列表
- 输入框
- 发送按钮
- 打开应用按钮

建议定位：

- 靠近桌宠显示
- 小尺寸浮层
- 默认无侧边栏
- 不进入复杂配置

### 2. 主应用首页

首页用于展示最核心信息。

建议包含：

- 当前桌宠状态
- 当前任务状态
- 最近任务摘要
- 常用入口卡片
- 运行状态概览

### 3. 对话页面

用于完整聊天与任务发起。

建议包含：

- 对话历史
- 输入框
- 发送按钮
- 当前任务状态
- 最近结果摘要

### 4. 模型配置页

用于管理模型相关设置。

首版可先只做占位或简单表单，建议字段包括：

- 当前使用模型
- 基础参数
- 响应风格
- 连接状态

### 5. 技能配置页

用于管理技能开关和可见配置。

首版建议包含：

- 技能列表
- 启用/禁用开关
- 技能描述
- 简单参数区

### 6. 系统设置页

用于管理应用级配置。

首版建议包含：

- 启动项
- 窗口行为
- 本地连接状态
- 日志级别
- 版本信息

### 7. 状态与日志页

这一页虽然不是用户问题里的显式要求，但建议首版预留，因为联调会大量依赖它。

建议展示：

- 当前连接状态
- 当前任务 ID
- 最新状态流转
- 最近错误信息

## 九、数据模型建议

Godot 前台先不要直接绑定复杂后端对象，建议只消费统一的轻量视图模型。

### 1. 桌宠状态模型

```json
{
  "pet_id": "pet_01",
  "display_name": "QQPAL",
  "state": "idle",
  "mood": "normal",
  "current_task_id": "",
  "status_text": "待机中"
}
```

### 2. 简洁对话框模型

```json
{
  "messages": [],
  "input_enabled": true,
  "status_text": "可以直接提问",
  "show_open_app": true
}
```

### 3. 主应用导航模型

```json
{
  "pages": [
    "home",
    "chat",
    "model_config",
    "skill_config",
    "system_settings"
  ],
  "active_page": "home"
}
```

### 4. 任务状态模型

建议沿用项目已有统一字段方向，至少保证以下字段可被 Godot 消费：

- `task_id`
- `user_input`
- `plan`
- `status`
- `result`
- `error`
- `timestamps`

### 5. UI 状态映射建议

| 业务状态 | 桌宠状态 | UI 文案建议 |
| --- | --- | --- |
| idle | idle | 待机中 |
| planning | planning | 正在思考 |
| running | running | 正在执行 |
| waiting_user | waiting_user | 等待你确认 |
| success | success | 已完成 |
| failed | failed | 执行失败 |

## 十、开发顺序

为了保证这版能快速落地，建议按以下顺序开发。

### 第一阶段：先起桌宠骨架

- 建立 `DesktopPet` 场景
- 完成悬停、左键、右键三种输入
- 做出静态桌宠显示
- 能弹出右键菜单
- 能弹出简洁版对话框

### 第二阶段：先起状态机

- 接入 `idle / hover / talking`
- 再接入 `planning / running / success / failed`
- 最后补 `waiting_user / paused`

### 第三阶段：起主应用壳子

- 建 `MainApp` 场景
- 搭顶部栏和侧边导航
- 接入首页、对话页、模型配置页、技能配置页、系统设置页
- 页面可以先是占位版

### 第四阶段：打通入口关系

- 左键打开简洁对话框
- 右键打开快捷菜单
- 快捷菜单点击“打开应用”
- 简洁对话框点击“打开应用”
- 主应用可切换页面

### 第五阶段：接入状态联动

- 使用 mock 状态流
- 将业务状态映射到桌宠表现
- 将任务状态显示到主应用
- 将最近一次任务结果显示到 UI

### 第六阶段：再补视觉体验

- 补状态气泡
- 补状态切换动效
- 调整浮层层级
- 优化窗口尺寸和交互细节

## 十一、联调边界

为了避免 Godot 前台写死业务逻辑，建议边界保持清晰。

### 1. Godot 负责什么

- 桌宠显示
- 左键和右键交互
- 菜单与窗口导航
- 状态展示
- 对话输入输出展示

### 2. Runtime 负责什么

- 本地状态管理
- 配置保存
- 当前任务同步
- 错误和日志上报
- 给 Godot 推送标准化状态

### 3. 云主 Agent 负责什么

- 任务理解
- 任务拆解
- 结构化计划返回
- 跨端协调

### 4. Godot 不建议直接负责什么

- 复杂任务编排
- 模型推理控制
- 执行引擎调度
- 业务结果持久化逻辑

## 十二、验收标准

这一版完成时，至少满足以下标准：

### 1. 桌宠层验收

- 能看到一只桌宠
- 桌宠能响应悬停
- 左键能打开简洁版对话框
- 右键能打开快捷菜单

### 2. 主应用层验收

- 菜单里的“打开应用”能正常打开主界面
- 主界面至少有首页、对话页、模型配置页、技能配置页、系统设置页
- 页面之间能正常切换

### 3. 状态机验收

- 至少支持 `idle / talking / planning / running / success / failed`
- UI 状态变化时，桌宠表现同步变化
- 错误状态可见

### 4. 演示链路验收

- 用户点击桌宠可以快速发起一次交互
- 用户通过右键菜单可以进入完整应用
- 主应用可以看到任务状态与基础配置页面
- 整体链路形成“桌宠入口 + 应用主界面”的最小闭环

---

## 十三、建议的首版结论

这版 Godot UI 最适合定义为：

> **一只桌宠 + 一套基础状态机 + 左键简洁对话框 + 右键快捷菜单 + 一个完整主应用入口。**

如果继续往下拆，最优先实现顺序建议是：

1. 桌宠显示与点击事件
2. 右键菜单
3. 左键简洁对话框
4. 主应用壳子
5. 状态机映射
6. 配置页面占位

这条路线最符合当前简单版目标，也最容易在 Godot 4.6 稳定版中快速起出一个能演示、能联调、能继续扩展的前台原型。
