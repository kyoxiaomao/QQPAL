# runtime-core 开发任务拆解清单

## 一、目标

这份清单用于把 `runtime-core` 从“最小 demo”推进到“可与 Godot 联调、可逐步接入真实执行器”的版本。

当前建议采用 **分阶段推进**，每一阶段都要求：

- 有明确输出物
- 有最小可验证结果
- 不一次做过多抽象

---

## 二、当前起点

当前已经完成：

- `runtime-core/` 目录骨架
- Python 标准库 HTTP 服务
- `GET /health`
- `POST /tasks`
- `GET /tasks/{task_id}`
- demo 状态机
- smoke_demo 验证脚本

也就是说，现在已经具备了 **最小本地 Runtime 雏形**，接下来应该进入“协议完善 + Godot 对接 + 真执行替换”阶段。

---

## 三、阶段拆解总览

```text
P1 协议固化
P2 Runtime 内部增强
P3 Godot 接入 Runtime
P4 状态与展示打通
P5 接入真实执行器
P6 工程化与稳定性补强
```

---

## 四、P1：协议固化

### 目标

先把 Runtime 对外协议和内部任务模型定稳，避免后续 Godot、Runtime、执行器各写一套。

### 任务清单

- 明确任务请求字段
- 明确任务响应字段
- 明确任务状态枚举
- 明确状态流转规则
- 明确错误返回格式
- 明确时间戳字段语义
- 明确 demo 字段与正式字段边界

### 建议输出物

- `docs/` 下新增协议文档
- `runtime-core` 内部任务模型字段表
- 状态流转图

### 验收标准

- Godot、Runtime、后续执行器都能用同一套字段
- 不再依赖口头约定

---

## 五、P2：Runtime 内部增强

### 目标

把当前“单文件 demo”升级成更稳的本地任务运行层。

### 任务清单

#### 1. 配置能力

- 支持从配置文件读取参数
- 支持环境变量覆盖配置
- 增加默认超时配置
- 增加日志级别配置

#### 2. 任务模型

- 增加 `scene_type`
- 增加 `source`
- 增加 `metadata`
- 增加 `retry_count`
- 增加 `final_status`

#### 3. 任务管理

- 增加任务列表查询接口
- 增加并发任务限制
- 增加取消任务能力
- 增加失败重试能力
- 增加任务超时处理

#### 4. 状态机

- 补充 `waiting_user`
- 补充 `paused`
- 补充异常分支
- 补充非法状态流转保护

#### 5. 日志与监控

- 记录任务开始日志
- 记录状态变化日志
- 记录错误日志
- 增加简单运行指标

### 建议输出物

- `task/manager.py` 增强版
- `state/machine.py` 增强版
- `monitor/` 下最小日志模块

### 验收标准

- 多次提交任务不崩
- 失败任务可定位原因
- 状态流转可追踪

---

## 六、P3：Godot 接入 Runtime

### 目标

让 `godot-app` 不再直接依赖 `MockTaskService`，而是接入 `runtime-core`。

### 任务清单

#### 1. 替换数据来源

- 保留 `MockTaskService` 作为回退方案
- 新增 `RuntimeTaskService`
- 将提交逻辑从 mock 切到 HTTP 请求
- 将查询逻辑从本地模拟切到任务轮询

#### 2. 主流程接入点

- 快捷对话框提交接入 Runtime
- 主窗口对话页提交接入 Runtime
- 结果统一回填到主窗口和快捷对话框

#### 3. 失败处理

- Runtime 不可用时显示错误
- 请求超时时显示错误
- 查询不到任务时显示错误

### 建议输出物

- `godot-app/scripts/services/RuntimeTaskService.gd`
- `Main.gd` 中切换服务调用逻辑

### 验收标准

- 在 Godot 中输入一句话，能通过 Runtime 返回任务状态
- 不再完全依赖 `MockTaskService`

---

## 七、P4：状态与展示打通

### 目标

把 Runtime 的标准状态完整映射到桌宠、快捷对话框、大窗口 UI。

### 任务清单

#### 1. 状态映射

- `planning` 对应思考态
- `running` 对应执行态
- `waiting_user` 对应提醒态
- `success` 对应完成态
- `failed` 对应失败态

#### 2. 文案统一

- 统一状态提示文案
- 统一错误提示文案
- 统一成功结果展示格式

#### 3. 历史记录

- 追加系统消息
- 追加用户消息
- 追加结果消息
- 追加失败消息

### 建议输出物

- Godot 状态映射表
- 前端展示文案规范

### 验收标准

- 状态变化时桌宠和主窗口展示一致
- 成功、失败、等待用户三类路径都能看到正确反馈

---

## 八、P5：接入真实执行器

### 目标

把当前 demo 状态机中的“假执行”逐步替换成真实执行能力。

### 任务清单

#### 1. 执行器适配层

- 在 `engine/` 下建立适配器接口
- 约定执行输入结构
- 约定执行输出结构
- 统一异常格式

#### 2. OpenClaw 接入

- 建立 OpenClaw 调用适配模块
- 处理启动、调用、结束
- 捕获执行错误
- 把执行结果回写到任务模型

#### 3. 分阶段替换

- 先替换成功路径
- 再替换失败路径
- 最后补暂停、等待确认、恢复逻辑

### 建议输出物

- `engine/openclaw_adapter.py`
- 任务执行协议文档

### 验收标准

- Runtime 能真正驱动执行器
- 返回结果不是假数据

---

## 十、建议开发顺序

建议严格按下面顺序推进：

### 第一优先级

- P1 协议固化
- P3 Godot 接入 Runtime

### 第二优先级

- P2 Runtime 内部增强
- P4 状态与展示打通

### 第三优先级

- P5 接入真实执行器
- P6 工程化与稳定性补强

原因很简单：

- **先协议**
- **再联调**
- **再替换执行**
- **最后补工程化**

---

## 十一、最值得先落地的 7 个具体任务

如果只看“下一步马上做什么”，我建议先做这 7 项：

1. 写一份正式的 Runtime 协议文档
2. 为任务模型补齐字段定义
3. 增加任务列表查询接口
4. 新建 Godot 侧 `RuntimeTaskService`
5. 让快捷对话框提交走 `POST /tasks`
6. 让主窗口对话页轮询 `GET /tasks/{task_id}`
7. 把 `MockTaskService` 改成可切换模式

---

## 十三、完成标志

当满足下面这些条件时，可以认为 `runtime-core` 进入下一阶段：

- Godot 不再只依赖 mock
- Runtime 可以接收真实输入
- Runtime 可以输出标准状态
- 主窗口和快捷对话框都能展示真实任务状态
- 任务链路可以被真实执行器替换

---

## 十四、一句话结论

`runtime-core` 下一步最重要的开发方向，不是继续扩 demo，而是 **先固化协议，再接入 Godot，再逐步替换真实执行器**。
