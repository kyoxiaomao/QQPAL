# Godot 三窗口桌宠改造计划

## 目标

把当前 `godot-app` 从“单主窗口原型”改造成你确认的最终路线：

- 1 个 Godot 项目
- 1 个 exe
- 3 个独立 Window
- 1 份共享状态服务

并满足以下核心体验：

- 桌宠窗口：透明背景、无边框、置顶、默认常驻右下角
- 应用窗口：普通独立 UI 窗口，按需打开
- 快捷对话窗口：独立小浮窗，左键桌宠时打开
- 三个窗口共享同一份状态数据与任务流转

## 当前现状

基于现有工程，当前结构仍是“一个主窗口里挂桌宠和应用 UI”：

- `project.godot` 的主窗口仍是普通 `1280 x 720` 窗口
- `scenes/Main.tscn` 里当前直接挂了 `DesktopPet` 和 `MainApp`
- `DesktopPet` 内部仍直接挂着 `QuickChatPanel` 和 `PetContextMenu`
- `MockTaskService.gd` 已经可以作为共享状态服务的雏形继续复用

这意味着后续改造的重点不是重做业务，而是把“单窗口场景结构”重构成“多窗口结构”。

## 设计结论

### 一、采用单项目 + 多窗口，而不是超大透明桌面场景

本次不采用“一个铺满全桌面的超大透明场景”方案，原因如下：

- 会引入整屏点击命中、穿透、焦点管理问题
- 桌宠、快捷对话、主应用在系统层面不是真正独立窗口
- 后续右下角停靠、多屏支持、任务栏行为更难控制

因此本次直接采用 Godot 4 的多窗口能力。

### 二、窗口职责拆分

#### 1. 桌宠窗口

- 只承载桌宠本体与右键菜单触发逻辑
- 窗口本身透明、无边框、置顶
- 启动时自动定位到右下角
- 默认可见
- 不承担完整主应用 UI

#### 2. 快捷对话窗口

- 独立小浮窗
- 左键桌宠时显示
- 包含输入框、发送按钮、状态文案、打开应用入口
- 默认隐藏
- 打开时尽量贴近桌宠窗口位置

#### 3. 主应用窗口

- 独立正常窗口
- 承载首页、对话、模型配置、技能配置、系统设置等完整页面
- 默认隐藏
- 由桌宠右键菜单和快捷对话窗口中的“打开应用”唤起

### 三、状态关系

- 共享状态服务只保留一份
- 桌宠状态机仍由统一控制器负责
- 桌宠窗口、快捷对话窗口、主应用窗口都只消费统一状态
- 所有任务发起入口统一汇总到共享任务服务

## 具体改造步骤

### 第一阶段：重构主场景为窗口调度器

目标：让 `Main.tscn` 不再直接承担单窗口 UI，而是成为全局调度入口。

实施步骤：

1. 保留 `Main.gd` 作为总控脚本
2. 将 `Main.tscn` 调整为“窗口管理根场景”
3. 在主场景中挂载：
   - 共享状态服务
   - 桌宠窗口
   - 快捷对话窗口
   - 主应用窗口
4. 改造当前事件连接方式，使总控脚本只负责：
   - 窗口显示 / 隐藏
   - 各窗口之间事件转发
   - 状态广播

验收结果：

- 主场景只做调度，不直接承担大面积 UI

### 第二阶段：把桌宠从“场景节点”改造成“独立桌宠窗口”

目标：实现真正的桌宠形态。

实施步骤：

1. 为桌宠新增独立 `Window` 容器
2. 把当前 `DesktopPet.tscn` 作为桌宠窗口的内容场景
3. 设置桌宠窗口属性：
   - transparent
   - borderless
   - always_on_top
   - 初始尺寸收缩到只包住桌宠
4. 在运行时开启透明背景
5. 启动时通过脚本把桌宠窗口定位到屏幕右下角
6. 清理当前桌宠里的开发态“大片灰底观感”
7. 评估并处理状态标签显示方式：
   - 开发期可保留调试开关
   - 默认桌宠正式态不显示大块调试文本

验收结果：

- 启动后只看到宠物本体
- 没有标题栏
- 背景透明
- 置顶显示在右下角

### 第三阶段：把快捷对话框从“桌宠内嵌面板”改造成“独立小浮窗”

目标：左键桌宠后不在同一个窗口里展开，而是单独弹出一个小窗口。

实施步骤：

1. 为快捷对话新增独立 `Window`
2. 将 `QuickChatPanel.tscn` 改造成适合放入独立窗口的内容场景
3. 调整显示逻辑：
   - 左键桌宠 -> 打开快捷对话窗口
   - 再次关闭时只隐藏快捷对话窗口
4. 设置快捷对话窗口属性：
   - 无需全透明，但应无多余大背景
   - 小尺寸
   - 默认隐藏
5. 打开时根据桌宠窗口坐标计算相邻位置
6. 保留“发送任务”和“打开应用”入口

验收结果：

- 左键桌宠后，出现一个独立小浮窗
- 关闭快捷对话不会影响桌宠窗口

### 第四阶段：把主应用改造成独立应用窗口

目标：主应用不再是当前主窗口中的一个控件，而是单独的应用窗口。

实施步骤：

1. 为主应用新增独立 `Window`
2. 将当前 `MainApp.tscn` 作为该窗口的内容场景
3. 保持主应用窗口为普通 UI 窗口：
   - 有正常标题区域或自定义面板皆可
   - 默认隐藏
4. “打开应用”统一改为：
   - 显示主应用窗口
   - 切换到指定页面
5. 主应用关闭后仅隐藏自身窗口，不影响桌宠与快捷对话

验收结果：

- 主应用可单独打开 / 关闭
- 不影响桌宠常驻

### 第五阶段：统一共享状态服务和消息流

目标：三窗口共享一份状态，不复制业务逻辑。

实施步骤：

1. 延续使用 `MockTaskService.gd` 作为共享状态服务核心
2. 梳理统一输入入口：
   - 桌宠右键菜单
   - 快捷对话窗口发送
   - 主应用聊天页发送
3. 梳理统一输出广播：
   - 桌宠状态变化
   - 快捷对话状态文案变化
   - 主应用首页与对话页状态同步
4. 保证 `PetStateController.gd` 仍是桌宠状态机单一来源
5. 把窗口事件与业务状态解耦，避免窗口互相直接硬引用

验收结果：

- 任一入口发起任务，都能驱动三窗口同步更新

### 第六阶段：调整当前场景边界与脚本职责

目标：让每个文件职责更稳定，便于后续继续扩展。

实施步骤：

1. `Main.gd`
   - 负责全局窗口调度
   - 负责共享服务接线
2. `DesktopPet.gd`
   - 只负责桌宠视觉与输入响应
   - 不再直接持有快捷对话面板实例
3. `MainApp.gd`
   - 只负责主应用页切换与展示
4. `QuickChatPanel.gd`
   - 只负责快捷对话 UI 逻辑
5. `PetContextMenu.gd`
   - 保持作为桌宠右键菜单触发器

验收结果：

- 不再出现“所有东西都塞在 DesktopPet 场景里”的结构

### 第七阶段：透明桌宠专项验证

目标：确保桌宠不是“看起来透明”，而是实际可用的透明悬浮窗。

实施步骤：

1. 验证桌宠窗口背景真实透明
2. 验证边框确实被移除
3. 验证窗口始终置顶
4. 验证桌宠点击区域仍准确
5. 验证 idle 动画透明边缘无异常底色
6. 验证右键菜单弹出位置正常

验收结果：

- 桌宠窗口达到“只看到宠物本体”的效果

### 第八阶段：最终联调与导出验证

目标：保证三窗口结构在运行和导出后都能工作。

实施步骤：

1. 在编辑器内运行验证三窗口
2. 验证桌宠窗口默认出现
3. 验证左键打开快捷对话窗口
4. 验证右键菜单操作
5. 验证打开应用显示主应用窗口
6. 验证关闭主应用不影响桌宠
7. 验证导出后行为与编辑器内一致

验收结果：

- 运行态和导出态都符合三窗口设计

- `godot-app/scripts/pet/QuickChatPanel.gd`

- `godot-app/scripts/pet/PetContextMenu.gd`
- `godot-app/scenes/app/MainApp.tscn`
- `godot-app/scripts/app/MainApp.gd`
- 可能新增用于多窗口承载的场景或脚本

说明：

- 由于根目录的 `GodotUi开发.MD` 已被删除，本轮不会恢复它
- 以 `godot-app` 下现有工程和脚本结构为准继续演进

## 关键实施原则

### 1. 桌宠窗口优先级最高

这轮不是单纯修 UI，而是优先保证：

- 透明
- 无边框
- 置顶
- 右下角常驻

### 2. 主应用与快捷对话都不能影响桌宠驻留

无论主应用是否打开，桌宠都必须独立存在。

### 3. 不拆成多个 exe

继续坚持：

- 单项目
- 单进程
- 多窗口

### 4. 保留现有 idle 资源接入方式

当前已存在：

- `assets/pets/animations/idle/frame_000001.png` 到 `frame_000049.png`

这轮不重做 idle 资源命名，只在窗口结构层面完成重构。

## 预期结果

改造完成后，最终运行体验应为：

1. 启动程序，只看到右下角透明桌宠
2. 桌宠无边框、置顶、透明背景
3. 左键桌宠，弹出独立快捷对话小窗
4. 右键桌宠，弹出菜单
5. 点击“打开应用”，显示独立主应用窗口
6. 三个窗口都共用同一份状态服务
7. 桌宠状态变化能同步反映到快捷对话和主应用
