# Godot UI 开发计划

## 背景

当前 `godot-app` 目录下已有：

- UI 设计文档：`GodotUi开发.MD`
- 桌宠 idle 帧资源：`assets/pets/animations/idle/frame_000001.png` 到 `frame_000049.png`

当前还没有实际的 Godot 工程骨架、场景文件、脚本文件和主应用 UI 实现，因此接下来要从“文档阶段”进入“最小可运行原型阶段”。

## 实施目标

基于 `GodotUi开发.MD`，落地一个可运行的 Godot 4.6 简单版原型，至少具备以下能力：

1. 启动后显示一只桌宠
2. 桌宠播放 idle 帧动画
3. 左键点击桌宠弹出简洁对话框
4. 右键点击桌宠弹出快捷菜单
5. 快捷菜单中的“打开应用”可打开主应用界面
6. 主应用界面包含基础页面壳子
7. 桌宠具备最小状态机和 mock 状态切换能力

## 实施范围

### 第一阶段：初始化 Godot 工程骨架

1. 创建 Godot 4.6 项目基础文件
2. 建立文档中约定的目录结构：
   - `scenes/app`
   - `scenes/app/pages`
   - `scenes/pet`
   - `scenes/common`
   - `scripts/app`
   - `scripts/pet`
   - `scripts/state`
   - `scripts/services`
3. 保留现有资源目录 `assets/pets/animations/idle`
4. 设置一个可直接运行的主场景入口

### 第二阶段：实现桌宠场景与 idle 动画

1. 创建 `DesktopPet.tscn`
2. 采用 2D 方案实现桌宠显示，优先使用 `AnimatedSprite2D + SpriteFrames`
3. 将 `assets/pets/animations/idle` 下的 49 张帧图导入为 idle 动画
4. 保证桌宠启动后即可循环播放 idle
5. 为后续状态扩展预留动画切换接口

### 第三阶段：实现桌宠交互层

1. 实现鼠标悬停反馈
2. 实现左键点击打开 `QuickChatPanel`
3. 实现右键点击打开 `PetContextMenu`
4. 菜单项至少包含：
   - 打开应用
   - 快速对话
   - 查看状态
   - 暂停/恢复桌宠
   - 退出
5. 将交互逻辑与表现节点适度分离，避免全部写入单脚本

### 第四阶段：实现主应用壳子

1. 创建 `MainApp.tscn`
2. 实现基础布局：
   - 顶部栏
   - 侧边导航
   - 页面容器
3. 创建占位页面：
   - `HomePage.tscn`
   - `ChatPage.tscn`
   - `ModelConfigPage.tscn`
   - `SkillConfigPage.tscn`
   - `SystemSettingsPage.tscn`
4. 让“打开应用”能正常显示主应用界面
5. 支持页面切换

### 第五阶段：实现最小状态机

1. 创建桌宠状态控制脚本
2. 首批接入以下状态：
   - `idle`
   - `hover`
   - `talking`
   - `planning`
   - `running`
   - `success`
   - `failed`
3. 其中只有 `idle` 先接真实动画资源，其余状态先用占位逻辑或文案反馈
4. 保持状态定义与 `GodotUi开发.MD`、`p0-协议同步清单.md` 一致
5. 为后续接入 `waiting_user` 和 `paused` 预留入口

### 第六阶段：加入最小 mock 联动

1. 创建一个简单的本地 mock 服务或状态驱动器
2. 让桌宠可通过按钮或测试入口模拟状态流转：
   - `idle -> talking -> planning -> running -> success`
   - `idle -> talking -> planning -> running -> failed`
3. 在 UI 中显示当前状态文案
4. 在主应用首页或状态区展示当前任务状态占位信息

### 第七阶段：验证与收尾

1. 在编辑器中验证项目可以打开
2. 验证主场景可运行
3. 验证 idle 动画正常播放
4. 验证左键和右键交互正常
5. 验证“打开应用”可进入主应用
6. 验证 mock 状态切换可见
7. 修正文档与实际目录、命名、状态差异

## 关键实现约定

### 技术选择

- 默认使用 GDScript 落地首版
- 默认使用 2D 节点而不是 3D
- 默认优先实现能跑通的最小结构，不先引入复杂插件

### 动画资源约定

- 当前真实资源路径：`assets/pets/animations/idle/`
- 当前真实帧文件命名：`frame_000001.png` 到 `frame_000049.png`
- 首版实现时以“读取现有真实资源”为准，不强行重命名资源

### UI 策略

- 简洁对话框先做轻量浮层
- 主应用页先做结构，不追求首版视觉完成度
- 复杂设置页先放占位内容，确保入口和布局先成立

4. 主应用壳子

## 风险与处理

### 风险 1：idle 图片尺寸或锚点不统一

处理方式：

- 先按现有资源导入
- 在场景中通过缩放、居中和偏移校正
- 如资源本身存在问题，再单独整理资源规范

### 风险 2：还没有 hover / talking / running 的真实动画资源

处理方式：

- 首版先让这些状态通过文案、颜色、轻微缩放或占位动画表现
- 后续资源补齐后再替换为正式帧动画

### 风险 3：没有现成 Runtime 接口

处理方式：

- 首版先用本地 mock 驱动
- 等 Runtime 接口稳定后再替换数据源

## 通过标准

当以下条件全部满足时，认为第一轮开发完成：

1. Godot 项目可打开并运行
2. 桌宠 idle 动画可循环播放
3. 左键可打开简洁对话框
4. 右键可打开菜单
5. “打开应用”可打开主应用
6. 主应用至少有基础导航与占位页面
7. 状态机和 mock 状态切换可见
