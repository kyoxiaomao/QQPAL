## 说明

这份清单服务于 QQPAL 的 P0 阶段，核心原则只有一句：

> **谁先真实用到，谁先定义；写进清单后，其他端按清单同步。**

也就是说：

- 不追求一开始把所有协议都定完
- 先在开发中暴露真实需要
- 谁先用到字段、状态、接口，谁先补进这份清单
- 一旦写进清单，就作为当前统一版本
- 后续如果调整，直接更新这份清单，不靠口头同步

## 二、当前模块职责

### Godot

- 负责：桌宠表现、左键简洁对话框、右键菜单、主应用页面、状态展示
- 不负责：任务编排、执行引擎调度、复杂业务持久化

### Runtime

- 负责：本地状态管理、任务进度同步、配置落地、向 Godot 输出标准化状态
- 不负责：前台 UI 表现、云端任务理解

### 云主 Agent

- 负责：任务理解、任务拆解、结构化计划返回、跨端协调
- 不负责：桌宠前台渲染、本地 UI 资源管理

### OpenClaw

- 负责：执行具体桌面任务
- 不负责：前台状态展示、主应用配置界面

### 当前状态解释

- `idle`
  - 系统空闲，没有任务在执行
  - Godot 可以表现为待机态

- `planning`
  - 用户已提交任务，系统正在拆解任务或生成执行计划
  - 当前主要对应云主 Agent 的规划阶段

- `running`
  - 执行计划已经生成，Runtime 或 OpenClaw 正在真正执行任务
  - Godot 可以表现为忙碌态

- `waiting_user`
  - 任务执行过程中需要用户补充信息或确认
  - Godot 可以表现为提醒态或提问态

- `success`
  - 任务已经完成，并且结果成功
  - 可以用于展示完成结果、成功提示或正向反馈

- `failed`
  - 任务已经结束，但执行失败
  - 可能来自规划失败、执行失败、超时或引擎异常

- `hover`
  - 鼠标悬停在桌宠上时的本地交互态
  - 这是 Godot 本地 UI 状态，不要求云端参与

- `talking`
  - 左键打开简洁对话框后的本地交互态
  - 这是 Godot 本地 UI 状态，不要求云端参与

- `paused`
  - 桌宠进入暂停或低活跃模式
  - 可以由 Godot 本地触发，也可以消费 Runtime 的暂停信号

## 五、当前状态流转

```text
idle
-> planning
-> running
-> waiting_user / success / failed
```

- 当前以哪一端定义为准：共享业务状态以 Runtime / 云主 Agent 协议为准，本地交互状态以 Godot 为准
- 当前还有哪些状态未统一：后续如果新增安装中、离线中、恢复中，再追加到状态清单
