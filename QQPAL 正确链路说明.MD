# QQPAL 正确链路说明

## 一、目的

这份文档用于统一 QQPAL 当前阶段的正确链路理解，避免后续在架构讨论、文档编写、接口设计中再次把：

- Godot
- 云端大模型 / 主 Agent
- runtime-core
- OpenClaw

这四者的职责混在一起。

---

## 二、一句话结论

QQPAL 当前阶段的正确链路不是：

```text
Godot -> Runtime -> 云端 -> 执行器
```

而是：

```text
Godot 负责 UI 与输入
云端大模型负责和用户交流、理解任务、决定是否执行
runtime-core 负责本地任务编排、排队、阻塞控制、执行器调度
OpenClaw 负责真正执行
```

---

## 三、核心角色定位

### 1. Godot

Godot 是前端 UI，不是任务中枢。

主要职责：

- 展示桌宠和主界面
- 承接用户输入
- 展示对话内容
- 展示任务状态和执行结果
- 展示本地 Runtime / 执行器状态

Godot 不负责：

- 自己理解复杂任务
- 自己决定任务是否执行
- 自己编排本地执行器

### 2. 云端大模型 / 主 Agent

云端是对话主脑和任务决策中心。

主要职责：

- 和用户交流
- 理解自然语言输入
- 判断当前是普通对话还是执行型任务
- 生成结构化任务
- 在需要时把任务下发给 runtime-core

### 3. runtime-core

runtime-core 不是聊天主脑，而是本地任务编排中枢。

主要职责：

- 接收云端下发的执行任务
- 维护本地任务队列
- 根据执行器状态决定是否立即下发任务
- 处理任务阻塞、排队、失败、恢复
- 汇总执行状态并提供本地状态查询接口

### 4. OpenClaw

OpenClaw 是执行器。

主要职责：

- 真正执行桌面任务
- 返回执行状态
- 返回执行结果、错误、日志、截图等信息

---

## 四、正确链路拆分

QQPAL 当前应该明确分成两条主链路：

- 对话链
- 执行链

---

## 五、对话链

### 正确链路

```text
Godot -> 云端大模型 -> Godot
```

### 含义

- 用户在 Godot 中输入一句话
- 这句话先发往云端大模型
- 云端大模型与用户进行对话、澄清、追问、确认
- 如果只是聊天，到这里就结束

### 关键结论

- 普通对话不应该先发给 runtime-core
- runtime-core 不是对话入口
- runtime-core 不负责承接所有用户聊天内容

---

## 六、执行链

### 正确链路

```text
Godot -> 云端大模型 -> runtime-core -> OpenClaw
```

执行状态回传链路：

```text
OpenClaw -> runtime-core -> 云端 / Godot
```

### 含义

- 用户输入先进入云端
- 云端判断该输入需要执行任务
- 云端生成任务并下发给 runtime-core
- runtime-core 根据本地执行器状态决定：
  - 立即执行
  - 进入队列
  - 等待前一个任务结束
  - 阻塞等待条件满足
- OpenClaw 执行后，把状态和结果回给 runtime-core
- runtime-core 再把本地状态暴露给 Godot

### 关键结论

- runtime-core 的核心价值是“本地调度”
- runtime-core 的存在原因不是替代云端对话
- runtime-core 要解决的是“执行器有限资源下的任务编排问题”

---

## 七、为什么必须有 runtime-core

如果没有 runtime-core，云端虽然能理解任务，但会遇到几个问题：

- 本地执行器是否空闲，云端不一定知道
- 多任务并发或阻塞时，需要本地判断
- 执行器失败、重试、恢复，更适合本地处理
- 安装状态、版本状态、心跳状态，更适合本地维护

所以 runtime-core 的存在，本质上是为了处理：

- 本地资源调度
- 执行器编排
- 本地状态汇总

---

## 八、Godot ↔ Runtime 的真实关系

Godot 和 runtime-core 之间确实保留：

```text
Godot ↔ Runtime：本地 HTTP API
```

但要注意，这条链路的含义不是：

```text
Godot 把所有任务都先发给 Runtime
```

而是更偏向：

- 读取本地执行状态
- 读取任务进度
- 读取执行器状态
- 读取安装状态
- 查询 Runtime 健康状态
- 在必要时显示本地执行结果

### 正确理解

- 对话主入口：云端
- 本地执行入口：runtime-core
- Godot 是 UI 和展示层

---

## 九、当前阶段建议通信关系

### 1. Godot ↔ 云端

建议承担：

- 用户对话提交
- 大模型回复内容
- 会话上下文同步
- 任务意图确认

### 2. 云端 ↔ runtime-core

建议承担：

- 任务下发
- 本地执行调度请求
- 本地任务状态同步
- 任务结果汇总

### 3. Godot ↔ runtime-core

建议承担：

- 本地执行状态读取
- 本地 Runtime 健康检查
- OpenClaw 安装状态查看
- 当前执行进度展示

---

## 十、最容易混淆的错误理解

### 错误理解 1

```text
Godot 的所有输入都发给 runtime-core
```

错误原因：

- 这会让 runtime-core 变成聊天主入口
- 与 runtime-core 的本地调度职责不匹配

### 错误理解 2

```text
runtime-core 负责和用户直接交流
```

错误原因：

- 用户交流主脑应该是云端大模型
- runtime-core 是本地编排层

### 错误理解 3

```text
Godot 和 runtime-core 是完全对等双向通信
```

错误原因：

- 两者并不是平级业务中心
- Godot 是 UI
- runtime-core 是本地执行编排层

---

## 十一、建议写入后续文档的标准表述

后续所有文档建议统一使用下面这段话：

> Godot 作为前端 UI，负责用户输入和状态展示；对话类请求优先发送到云端大模型，由云端负责交流、理解和任务决策；当任务需要本地执行时，再由云端把结构化任务下发给 runtime-core；runtime-core 根据本地执行器状态进行任务排队、阻塞控制和调度，并驱动 OpenClaw 执行；执行状态由 runtime-core 汇总后提供给 Godot 展示。

---

## 十二、一句话总结

QQPAL 的正确链路是：

**Godot 负责前台，云端负责交流与决策，runtime-core 负责本地编排，OpenClaw 负责执行。**
