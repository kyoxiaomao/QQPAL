# QQPAL 演示版开发时间表

## 说明

本文档基于当前已收敛的开发范围制定：

- 仅做 **Godot 桌宠/UI**
- 仅做 **本地 Runtime**
- 仅做 **云端主 Agent**
- 仅做 **1 个龙虾引擎 / OpenClaw 对接**
- 仅做 **PC 主链路演示版**
- 默认 **你 1 个人总控 + 多台电脑 + 多个 Trae Solo 账号并行开发**

目标是在最短时间内完成一版可以演示的 QQPAL 主链路：

> Godot 提交任务 → 云主 Agent 拆解任务 → Runtime 接收执行计划 → 龙虾引擎执行 → 状态回传 → Godot 宠物状态联动展示

---

## 一、为什么改成按优先级排

如果继续按天数看，会出现一个问题：

- 容易以为所有事情是并列推进的
- 容易先做界面细节，再补协议和状态
- 容易出现多线同时开工，但底层字段还没定死
- 容易在第 3～5 天反复改接口，联调成本变高

所以这份时间表改成：

- **先看优先级**
- **再看先后依赖**
- **最后再映射回大致时间**

核心原则只有一句：

> 先做“不做它，后面全部会返工”的事；再做“能跑起来”的事；最后做“让演示更完整、更稳定”的事。

---

## 二、优先级总览

| 优先级 | 阶段名称 | 为什么最先做 | 做完后的结果 |
| --- | --- | --- | --- |
| P0 | 定边界与统一协议 | 不先定死，Godot、Runtime、云端会各写各的 | 后续三条主线能按同一套字段开发 |
| P1 | 起最小骨架与最小闭环 | 没有骨架就无法验证主链路是否真的通 | Godot、Runtime、云端都能独立启动并形成假闭环 |
| P2 | 接入真实执行链路 | 只有 mock 不算真正可演示能力 | OpenClaw 至少完成 1 次真实任务执行 |
| P3 | 补全展示与异常分支 | 只展示成功链路，演示会显得很虚 | 成功、失败、等待、安装等状态都能讲清楚 |
| P4 | 联调打磨与冻结 | 不冻结就会一直改，无法形成演示版 | 得到可录屏、可复现、可现场展示的版本 |

---

## 三、推荐先后顺序

| 顺序 | 必做阶段 | 是否可并行 | 依赖关系 |
| --- | --- | --- | --- |
| 1 | P0 定边界与统一协议 | 低 | 必须最先完成 |
| 2 | P1-A 起三端最小骨架 | 高 | 依赖 P0 |
| 3 | P1-B 打通假闭环 | 中 | 依赖 P1-A |
| 4 | P2 接入真实 OpenClaw 执行 | 中 | 依赖 P1-B |
| 5 | P3 补宠物状态、安装页、异常分支 | 高 | 依赖 P2 主链路已通 |
| 6 | P4 联调、彩排、冻结 | 中 | 依赖前面全部基本完成 |

可以把它理解成：

1. 先定规则  
2. 再把三端跑起来  
3. 再把假链路跑通  
4. 再替换成真实执行  
5. 再把演示该有的状态补齐  
6. 最后只做修正，不再扩需求

---

## 四、P0：定边界与统一协议

这是最高优先级。这个阶段不追求“功能完成”，只追求“后面所有人都不会乱写”。

### 这个阶段必须做哪些事情

#### 1）定唯一演示场景

为了保证演示版一定能做出来，只选 **1 个任务场景**。

| 方案 | 演示任务 | 优点 | 风险 |
| --- | --- | --- | --- |
| 方案 A | 网页任务：打开网页、登录、上传、提交 | 最直观，最适合演示链路完整性 | 依赖网页流程稳定 |
| 方案 B | 文档任务：读取材料、生成摘要、输出结果 | 更贴近办公场景 | 结果展示较抽象 |
| 方案 C | 消息任务：打开应用、发送固定消息 | 演示速度快 | 视觉冲击力略弱 |

**推荐优先选择：方案 A，网页任务。**

#### 2）定系统边界

必须明确这版只保留以下能力：

- Godot 负责桌宠与展示层
- 云主 Agent 负责接任务、拆计划、回结构化结果
- Runtime 负责接计划、管状态、调引擎
- OpenClaw 只接入 1 条真实执行路径
- 先不做商业化后台、复杂权限、移动端主执行、多场景任务库

#### 3）定任务模型

先把任务对象统一，不要求字段最终完美，但必须统一。

至少定清楚这些内容：

- 任务主键怎么生成
- 用户原始输入是什么
- 云端拆解结果长什么样
- Runtime 收到的执行计划长什么样
- 执行结果与失败结果如何表达
- Godot 查询状态时看到哪些字段

建议最少统一以下几类字段：

- `task_id`
- `user_input`
- `plan`
- `status`
- `result`
- `error`
- `timestamps`

#### 4）定状态枚举

至少要定两层状态：

- **执行层状态**：`queued / running / succeeded / failed`
- **展示层状态**：`idle / planning / running / success / failed`

同时定清楚：

- 哪个状态由云端写
- 哪个状态由 Runtime 写
- Godot 是轮询还是事件订阅
- 失败时返回什么文案给 UI

#### 5）定接口协议

至少把这三类接口定死：

- Godot → 云端：提交对话与任务意图
- 云端 → Runtime：下发结构化执行任务
- Godot → Runtime：读取本地状态、执行进度、安装状态

接口草案里必须写清：

- 请求字段
- 响应字段
- 错误结构
- 状态字段取值
- 最小 mock 示例

#### 6）定目录和责任边界

这个动作非常重要，否则第二天大家起项目骨架时目录会乱。

要明确：

- Godot 工程目录谁负责
- Runtime 工程目录谁负责
- 云主 Agent 工程目录谁负责
- 联调脚本和 mock 数据放哪里
- 协议文档由谁维护

### 这个阶段的产出

- 演示场景说明
- 任务模型文档
- 状态枚举表
- 接口协议草案
- 三端职责边界说明

### 这个阶段的验收标准

- 后续三条线都能按统一字段开发
- 不再出现“这个状态叫什么”这种讨论
- 不再出现“这个字段到底谁返回”这种讨论
- 第 2 阶段可以直接开工，不需要重新定协议

## 五、P1：起最小骨架与打通假闭环

这是第二优先级。目标不是“做完整”，而是“先证明主链路跑得起来”。

### P1-A：起三端最小骨架

#### Godot / UI 线要做什么

- 初始化 Godot 项目
- 建主界面骨架
- 至少留出桌宠区、任务输入区、状态区、结果区
- 先放静态占位，不追求美术和动效
- 预留宠物状态展示位

#### Runtime 线要做什么

- 初始化 Runtime 项目
- 做健康检查接口
- 做任务接收接口骨架
- 做状态查询接口骨架
- 建最小状态机壳子

#### 云主 Agent 线要做什么

- 初始化云服务
- 做任务提交接口
- 做 mock 拆解接口
- 先返回固定结构化计划
- 不急着做复杂推理和复杂编排

#### 联调线要做什么

- 准备统一 mock 数据
- 准备接口示例请求
- 准备最小联调脚本
- 验证三端字段是否一致

### P1-A 的验收标准

- Godot 能打开并看到基础界面
- Runtime 能启动并返回健康状态
- 云主 Agent 能收任务并返回 mock 结果
- 联调线能用一组样例数据串三端

### P1-B：打通假闭环

这一步要把“看起来是一个系统”先做出来。

#### 这一阶段详细要做什么

- Godot 可以发起一次真实任务提交动作
- 云主 Agent 返回结构化计划和 `task_id`
- Runtime 能接到计划并写入状态流转
- 状态先用模拟执行，不接真实 OpenClaw
- Godot 不再只显示假页面，而是显示来自 Runtime 的真实状态
- 最终能展示提交中、执行中、完成态

#### 这个阶段为什么很关键

因为它会最早暴露三类问题：

- 协议字段不统一
- 状态流转设计不完整
- UI 与 Runtime 对状态理解不一致

### P1 的总验收标准

- 第一次完整假闭环成功
- 可以现场演示一条“提交任务到完成”的链路
- 即使后面不接真实引擎，也已经有一个能讲故事的 Demo 壳子

---

## 六、P2：接入真实 OpenClaw 执行链路

这是第三优先级，也是从“像 Demo”变成“真 Demo”的关键阶段。

### 这个阶段详细做什么

#### Runtime 侧

- 接 OpenClaw 的启动方式
- 接任务调用方式
- 接状态查询方式
- 把真实执行结果映射到 Runtime 统一状态机
- 保留 mock 执行作为降级开关

#### 云主 Agent 侧

- 调整计划结构，确保 Runtime 真能消费
- 去掉只为 mock 服务的多余字段
- 保证计划字段稳定，不再频繁改名

#### Godot 侧

- 展示真实执行中的进度和结果
- 至少能区分“规划中”和“执行中”
- 给结果区预留真实输出展示位

### 这个阶段最关键的控制点

- 只打通 **1 个真实场景**
- 不要同时接多个引擎
- 不要同时做多个执行模版
- 先保证“一次真实调用成功”比“多种能力都半成品”更重要

### 这个阶段的验收标准

- OpenClaw 至少被真实调用一次
- Runtime 能拿到真实执行反馈
- 至少 1 条任务不再依赖模拟执行

---

## 七、P3：补全展示、安装与异常分支

这是第四优先级。主链路通了以后，演示体验的可信度主要靠这个阶段补上。

### 1）宠物状态映射

把技术状态转成用户能看懂的宠物表现。

至少要覆盖：

- `idle`
- `planning`
- `running`
- `success`
- `failed`

同时明确：

- 每个状态对应什么 UI 表现
- 是否需要文案提示
- 结果区和宠物状态是否同时变化

### 2）安装引导与引擎就绪页

因为演示时一定会被问到“引擎从哪来”，所以不能只做执行成功页。

至少要补：

- 未安装页
- 安装中页
- 已安装可执行状态
- Runtime 侧的引擎检测骨架
- 云端侧的版本/能力信息说明

### 3）异常分支

演示系统如果只会成功，会显得很假。

至少补这几类分支：

- Runtime 未就绪
- OpenClaw 调用失败
- 任务超时
- 云端拆解失败
- 用户侧等待确认

UI 至少能展示：

- 失败提示
- 重试提示
- 当前阻塞原因
- 最终失败结果

### 这个阶段的验收标准

- 成功和失败都能展示
- 演示链路完整，不需要靠口头补解释
- 宠物状态会随真实任务变化，而不是手动切换

---

## 八、P4：联调、彩排与冻结

这是最后优先级，但不能跳过。这个阶段不再扩功能，只做收口。

### 这个阶段详细做什么

#### 联调打磨

- 修界面跳转问题
- 修状态闪烁问题
- 修字段不一致问题
- 修接口兼容问题
- 修日志与错误提示问题

#### 彩排

- 至少完整跑 2 轮以上
- 一轮走成功链路
- 一轮走失败链路
- 检查现场演示是否存在等待过长、路径过深、文案不清的问题

#### 冻结

- 冻结界面结构
- 冻结任务输入输出格式
- 冻结状态流
- 冻结演示文案
- 输出录屏脚本或演示操作手册

### 这个阶段的验收标准

- 可录制
- 可复现
- 可现场展示
- 演示过程中没有明显断链

## 九、多台电脑与多账号分工表

| 电脑 / 账号 | 建议负责内容 | 在优先级体系中的主要职责 | 每阶段交付重点 |
| --- | --- | --- | --- |
| 电脑 A / 账号 A | Godot / UI 线 | 把技术状态转成可演示界面 | 主界面骨架、状态展示、结果页、安装页、失败页 |
| 电脑 B / 账号 B | Runtime 线 | 建本地状态机并对接真实引擎 | 健康检查、任务流转、OpenClaw 接入、统一状态输出 |
| 电脑 C / 账号 C | 云主 Agent 线 | 提供任务接收、拆解和计划下发 | 提交接口、mock 拆解、真实计划结构、失败返回 |
| 电脑 D / 账号 D | 对接 / 联调线 | 保证协议一致并做集成验证 | mock 数据、联调脚本、验收脚本、状态映射检查 |
| 你本人 | 总控与集成 | 决定先后级、冻结协议、验收演示链路 | 定边界、排优先级、收口决策、彩排冻结 |

## 十、如果再映射回时间，大致应这样理解

虽然现在按优先级组织，但仍然可以粗略映射回时间。

| 优先级阶段 | 大致落点 | 说明 |
| --- | --- | --- |
| P0 定边界与统一协议 | 第 1 天 | 必须一次定清主链路规则 |
| P1 起骨架与假闭环 | 第 2～7 天 | 先把系统跑起来，比先做细节更重要 |
| P2 接真实执行 | 第 8～10 天 | 只攻克 1 条真实任务链路 |
| P3 补安装、异常、展示 | 第 10～12 天 | 让演示更完整、更可信 |
| P4 联调冻结 | 第 13～14 天 | 不再扩需求，只做收口 |

---

## 十一、最终执行建议

### 最重要的执行顺序

1. **先冻结协议和状态**
2. **再搭三端骨架**
3. **再跑通假闭环**
4. **再接 1 条真实 OpenClaw**
5. **再补异常、安装、宠物表现**
6. **最后只做联调和彩排**

### 最容易犯的错误

- 先做 UI 细节，后定协议
- 同时做多个演示场景
- 真实执行还没通，就开始补太多页面
- 主链路还不稳，就提前扩更多能力

### 一句话结论

> 这版 QQPAL 演示项目最合理的推进方式，不是“按日历平均铺开”，而是严格按优先级推进：**先定规则，再做骨架，再通闭环，再接真实执行，最后补演示完整度与冻结版本。**
